# Dictionnaire de données — Préparation modèles Django

> Référentiel unique consolidé à partir des 7 modules ER/Classes.
> Objectif : servir de base directe à l'écriture des `models.py` Django (les tables PostgreSQL seront générées par les migrations Django, pas créées manuellement).

**Conventions de typage Django utilisées :**
| Type diagramme | Type Django recommandé |
|---|---|
| UUID (PK) | `models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)` |
| int (PK) | `models.AutoField` (implicite, `id` par défaut) |
| string | `models.CharField(max_length=...)` |
| text | `models.TextField()` |
| bool | `models.BooleanField(default=...)` |
| decimal | `models.DecimalField(max_digits=10, decimal_places=2)` |
| date | `models.DateField()` |
| datetime | `models.DateTimeField()` |
| json | `models.JSONField()` |
| FK | `models.ForeignKey('Cible', on_delete=models.PROTECT/CASCADE/SET_NULL, related_name=...)` |

## Points d'attention pour l'implémentation Django

1. **App structure recommandée** : découper en apps Django alignées sur les modules (`users`, `catalog`, `orders`, `payments`, `monetization`, `shopping`, `analytics`) pour garder la même clarté que les diagrammes.
2. **UUID partout où c'est une entité "métier" exposée en API** (évite l'énumération d'IDs) ; `int` autoincrement réservé aux tables techniques/pivot.
3. **`payee_id` et `subscriber_id`** ne sont pas des FK Django classiques (association polymorphe) → utiliser soit `GenericForeignKey` (contenttypes), soit une validation au niveau du serializer/service.
4. **Champs `status`** : à convertir en `choices` (via `TextChoices`) plutôt que string libre, pour chaque entité concernée (Order, Payment, Delivery, Product, Subscription...).
5. **on_delete** à définir soigneusement : `PROTECT` pour les relations financières (Order, Payment), `CASCADE` pour les enfants de composition (OrderItem, CartItem, ProductImage), `SET_NULL` pour les relations optionnelles.
